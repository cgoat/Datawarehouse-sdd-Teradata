Building a Data Warehouse on Teradata using Spec Driven Development

Four short markdown files told the build what to do. The interesting story is how strictly that worked — and where the specs paid for themselves.

A walkthrough of layered ingestion, dbt on Teradata, Great Expectations, dynamic FastLoad selection, and what happens when you apply a "performance tuning" that makes things slower — with each decision traced back to the spec that demanded it.

Spec Driven Development, briefly

Spec Driven Development (SDD) is an unfussy idea: before you write code, write down — in plain English, in small files — what you're building and why. Not requirements documents, not Confluence pages, not Jira epics. Four short markdown files in the repo root, each answering one question:

  • mission.md — why does this exist? One-line pitch, target audience, goals, non-goals, personas.
  • roadmap.md — what ships, in what order? Vertical phases, each demoable on its own.
  • tech-stack.md — what tools, on purpose? Runtime, framework, data layer, deployment.
  • README.md — what's already built? The current state, never the future.

These aren't decoration. Every phase below started by re-reading the relevant spec line, and every contentious choice (do we drop bad rows or tag them? do we skip inventory for the demo? do we change a primary index?) was settled by checking what the specs said. The specs are short on purpose — under 200 lines combined — so they get read.

The whole repo lives at github.com/cgoat/Datawarehouse-sdd-Teradata, with the four spec files at the top of the tree. The rest of this article walks through the build with each section anchored to the spec line that drove it.

The specs, abbreviated

From mission.md:

Goals: Timely — provide data for timely decision making. Quality — data that adheres to the business rules.
Personas: Business Analyst (analyses and reports), CEO (reviews), Reliability Engineer (monitors pipelines for failures).
Non-goals: SaaS infrastructure; Reporting tool.

From roadmap.md:

Small, vertical slices. Each phase is shippable and demo-able on its own. Phase 0 — Bronze. Phase 1 — Silver (with anomaly tables). Phase 2 — Gold (integrated marts). Phase 3 — Performance tuning (review long-running queries, EXPLAIN, apply changes, show improvement).

From tech-stack.md:

DBT for transformations. Python for ingestion and API. PHP for dashboards. Open source for orchestration and DQ. Teradata as the warehouse. Bronze/Silver/Gold layers.

Three roles, two goals, four phases, six tools. That's the whole brief. Source data: TPC-DS scale factor 1 — 25 pipe-delimited files, ~1.2 GB, three sales channels (store, catalog, web), three corresponding returns tables, a fistful of dimensions. Target: a real Teradata 20 instance.

Phase 0 — Bronze, the loader story

Spec line: roadmap.md Phase 0 — "ingest using dbt framework into a raw zone (Bronze); schedule the ingestion as a batch job; perform data quality checks and generate a dashboard."

The first instinct on any Python-to-Teradata pipeline is the same: cursor.executemany("INSERT ... VALUES (?, ?, ?)", chunk) in a loop over pandas.read_csv(chunksize=10_000). It works. It's also genuinely slow.

My first end-to-end Bronze load took just over 17 minutes for ~7.8 M rows across 24 tables (skipping inventory's 11.7 M for the demo). The biggest single file, store_sales.dat (385 MB / 2.88 M rows), accounted for the bulk. Each batch was a parse-prepare-execute round-trip; with ~280 round-trips per fact table over a VM network, that adds up.

The right tool for the volume

Teradata has its own toolbox for bulk loads: FastLoad, MultiLoad, TPump, and TPT (Teradata Parallel Transporter, a meta-tool that drives the others). For a Python project where I'd rather not shell out to a separate CLI, the modern answer is the FastLoad escape that ships with the teradatasql driver itself: 

cur.execute("{fn teradata_try_fastload}")        # arms the connection
cur.executemany(
    "INSERT INTO DW_BRONZE.store_sales (...) VALUES (?, ?, ...)",
    rows,
)

But FastLoad has overhead: it opens a separate session, negotiates per-AMP delivery, then commits at end-of-statement. For a 12-row dimension like store, that's all cost and no benefit. So I made the loader pick: 

FASTLOAD_MIN_BYTES = int(os.getenv("FASTLOAD_MIN_BYTES", "1000000"))  # 1 MB

def pick_tool(path: Path) -> str:
    return "fastload" if path.stat().st_size >= FASTLOAD_MIN_BYTES else "executemany"

And — important for the dashboard — the chosen tool is recorded in DW_DQ.run_log.tool alongside the row count and duration. The Reliability Engineer can see at a glance which path each table took.

The bug that taught me about FastLoad sessions

My first attempt to wire in the picker failed instantly:

[Error 2636] catalog_page must be empty for Fast Loading.

But I had just truncated catalog_page. What gives?

Each call to cur.executemany() with the FastLoad escape opens its own FastLoad session and commits at the end. My loader still called read_csv(chunksize=10_000) and looped over chunks. The first chunk opened a session, inserted 10,000 rows, committed. The second chunk opened a fresh session — and FastLoad's "table must be empty" check now saw 10,000 rows. Boom.

The fix isn't elegant but it's the right one: if the picker says FastLoad, materialize the whole file and call executemany once. For SF1's largest file (store_sales at 2.88 M rows), that's ~700 MB of Python memory — fine on any modern machine. The chunked-streaming path is preserved for the executemany case where it still matters.

Total Bronze load went from ~17 min to ~6 min, all driven by a 5-line picker.

Phase 1 — Silver, anomaly sidecars beat data loss

Spec line: roadmap.md Phase 1 — "while moving the data clean the data per the business rules. Move any anomaly data to the anomaly tables." This is where the Quality goal in mission.md gets its teeth.

Silver is where business rules show up. Quantities should be positive, FKs should resolve, dates shouldn't be null. The spec is explicit: move anomaly rows to anomaly tables. That single line rules out the easy cheat — WHERE quantity > 0 filters and a shrug — and forces a deliberate sidecar pattern. The right move is to tag and route.

Every union of channel facts goes through one intermediate model that adds an _anomaly_reason column: 

SELECT
    u.*,
    CASE
        WHEN item_sk IS NULL                             THEN 'missing_item_sk'
        WHEN sale_order_id IS NULL                       THEN 'missing_order_id'
        WHEN sold_date_sk IS NULL                        THEN 'missing_sold_date'
        WHEN quantity IS NOT NULL AND quantity <= 0      THEN 'non_positive_quantity'
        WHEN sales_price IS NOT NULL AND sales_price < 0 THEN 'negative_sales_price'
        WHEN net_paid IS NOT NULL AND net_paid < 0       THEN 'negative_net_paid'
        ELSE NULL
    END AS _anomaly_reason
FROM ( /* UNION ALL of stg_store_sales, stg_catalog_sales, stg_web_sales */ ) u

Then two models read from the same intermediate:

  • fact_sales — WHERE _anomaly_reason IS NULL
  • fact_sales__anomalies — WHERE _anomaly_reason IS NOT NULL

TPC-DS at SF1 generates a deliberate ~2–3% of dirty rows. Our pipeline tagged exactly that:

TableRows
fact_sales (clean)4,903,874
fact_sales__anomalies137,462 (2.7%)
fact_returns (clean)490,205
fact_returns__anomalies13,139 (2.6%)

The Silver dashboard shows the breakdown by reason — which is the actually-useful question for a Reliability Engineer ("are we suddenly dropping 30% of orders for missing item_sk?"). Silent filters can't answer that.

The dbt-on-Teradata sharp edges

dbt-teradata works, but it has its quirks. A short tour of the ones I hit, all of which look like compiler errors and aren't:

Reserved-word column names

Teradata 20 reserves more words than I expected: date, year, class. My first dim_date aliased columns as date and year ("of course"); both failed parsing in CREATE TABLE AS SELECT with cryptic "expected something between AS and 'date'". The fix is mechanical (calendar_date, calendar_year) but you have to fix every downstream reference too. A grep saved me; the over-eager replace_all bit me.

Two-letter table aliases that aren't aliases

cd.cd_gender — innocent, right? Not on this Teradata. The two-letter alias cd after a comma in a SELECT list trips the parser. cdem.cd_gender works. Same story with cs as a CTE alias. Use three-letter aliases and stop arguing with the parser.

NTILE doesn't exist on this build

NTILE(5) OVER (ORDER BY x) is the textbook quintile. On this Teradata it raises "expected something between '(' and the integer '5'". The replacement is straightforward but worth saving as a macro: 

{% macro quintile(expr, direction='ASC') %}
CAST(CEILING(
    5.0
    * CAST(ROW_NUMBER() OVER (ORDER BY {{ expr }} {{ direction }}) AS FLOAT)
    / NULLIF(CAST(COUNT(*) OVER () AS FLOAT), 0)
) AS INTEGER)
{% endmacro %}

The FLOAT casts matter: integer arithmetic underflows to 0 for the smallest row when the dataset is large. I learned that when 989 customers landed with r_score = 0 and broke the test.

View chains across databases want WITH GRANT OPTION

My first silver/stg_* models were views; silver/intermediate/int_* were views built on top of those. Creating int_sales_unioned in DW_SILVER that referenced staging views (which referenced DW_BRONZE) tripped Teradata's transitive-permission check: the creator needs SELECT WITH GRANT OPTION on every link in the chain.

Three options: grant the privilege (denied — sensible default in shared environments), make the staging tables (extra disk, GRANT issue moves), or make the staging models ephemeral CTEs. The ephemeral path is best for this stack: dbt inlines staging into the consumer's SQL, the consumer is a CREATE TABLE AS SELECT that only needs plain SELECT, and there's no extra disk. 

# dbt_project.yml
silver:
  +schema: DW_SILVER
  stg:          { +materialized: ephemeral }
  intermediate: { +materialized: ephemeral }
  dim:          { +materialized: table }
  fact:         { +materialized: table }
  anomalies:    { +materialized: table }

One catch: dbt wraps ephemeral CTEs in a top-level WITH. If your intermediate model also starts with WITH, you get nested CTEs and Teradata declines ("WITH clause not supported within views"). Rewrite the intermediate as a plain SELECT ... FROM (subquery) and you're done.

+schema: SILVER doesn't mean what you think

In dbt-teradata, +schema defaults to a suffix applied to target.schema. So +schema: SILVER with target.schema = DW_SILVER writes to DW_SILVER_SILVER — except the cluster I was on already had a database called SILVER with 60 MB allocated, and dbt happily wrote into that, until it ran out of room. The fix is a one-liner macro override that treats +schema as absolute: 

{% macro generate_schema_name(custom_schema_name, node) -%}
    {%- if custom_schema_name is none -%}
        {{ target.schema }}
    {%- else -%}
        {{ custom_schema_name | trim }}
    {%- endif -%}
{%- endmacro %}

Data quality: dbt tests + Great Expectations, layered

Two complementary mechanisms, applied per layer:

  • dbt tests for structural invariants — not_null on PKs/FKs, unique on dimension keys, relationships on fact-to-dim joins, accepted_values on enum columns.
  • Great Expectations for distributional/business checks — value ranges, value sets with tolerances (mostly: 0.999), uniqueness on samples — and for persisting results to DW_DQ.dq_result so the dashboard can render PASS/FAIL trends per batch.

The combined coverage on the validated run was 53 dbt tests + 42 GE expectations across Bronze, Silver, and Gold. All pass except one I removed deliberately — and it shows up bright red on the dashboard:

The CHAR(1) padding trap. The teradatasql driver returns CHAR(1) values padded to display width, so customer.c_preferred_cust_flag comes back as 'Y ' (length 2). Any value_set: ['Y', 'N'] expectation fails on every row. The clean form lives in dim_customer.is_preferred_customer as 0/1. Test the cleaned column; let the raw one alone.

Phase 2 — Gold, four marts the personas asked for

Spec line: README.md stakeholder section — "Mrs. Business Analyst wants data in the consumption layer to do analysis." Specifically: mart_sales_performance, mart_customer_360, mart_product_analytics, mart_channel_comparison.

This is the SDD payoff in its purest form: the BA persona named four marts, the roadmap reserved Phase 2 for them, and the implementation is four .sql files. mart_customer_360 bundles RFM segmentation (Champion / Loyal / At Risk / Lost / New / Inactive / Potential) with lifetime value and preferred channel — one row per customer. mart_sales_performance is daily by channel; mart_channel_comparison is monthly with share-of-total. mart_product_analytics ranks 9,000 current items by revenue with returns rate and top channel.

No marts beyond what the spec named. Non-goal discipline is the same lever as goal discipline.

Phase 3 — performance tuning, with a humbling result

Spec line: roadmap.md Phase 3 — "review the long running queries; review the explain plan; make necessary changes (including the database settings) to improve based on the plan; show the improvement on the dashboard."

That last clause — show the improvement on the dashboard — is the SDD discipline that makes this section honest. The spec doesn't say "show improvement if it works." It says show it. So when the tuning made three of five queries slower, the dashboard had to render that result, in red, where the Reliability Engineer would see it. Phase 3 is where most "we tuned and got 5× faster!" articles end. This one isn't going to.

The harness

Five queries chosen to stress different access patterns: a full-scan group-by, a high-cardinality grouping, two FK-join aggregations, and a date-range filter. Each gets a unique comment per run so result-cache reuse can't cheat the numbers. Three iterations per phase. Median is what gets reported. 

# perf/queries.py — abbreviated
QUERIES = {
    "annual_sales_by_channel":   "...full scan + group by year/channel...",
    "top_25_customers":          "...group by customer_sk + limit...",
    "sales_by_category":         "...join dim_item + group by category...",
    "daily_sales_one_month":     "...filter Jan 2002 + group by date...",
    "returns_rate_by_brand":     "...sales LEFT JOIN returns + by brand...",
}

The tunings

Two:

  1. Comprehensive COLLECT STATISTICS on join columns, GROUP BY columns, and filter columns across every Silver and Gold table — single columns plus a few well-chosen combinations like (item_sk, sold_date_sk).
  2. Recreate fact_sales with PARTITION BY RANGE_N(sold_date_sk EACH 365) — a year-bucketed PPI in theory perfect for the daily-Jan-2002 query. To enable the partition, the primary index moved to (item_sk, sale_order_id).

What the dashboard told us

COLLECT STATISTICS earned its keep on the multi-join: returns_rate_by_brand joins fact_sales to dim_item and fact_returns. Better cardinality estimates let the optimizer pick smaller intermediate spool, and the query got 29% faster.

The PPI didn't help daily_sales_one_month at all — and this is the subtle part. The query filters on dim_date.calendar_year = 2002, not on fact_sales.sold_date_sk. For partition pruning to kick in, the optimizer would have to push the predicate from the dim through the join to the fact. With a 2-second baseline already, there wasn't much room and the predicate didn't propagate the way I'd hoped.

The two regressions — the queries that grouped by category or by year/channel — are about the primary index change. Before, fact_sales had the default PI dbt picks. After, I'd moved PI to (item_sk, sale_order_id) to enable PPI. Group-by aggregations now require fact_sales rows to be redistributed across AMPs by the grouping column — and that's pure cost.

The honest takeaway. On Teradata, primary index choice dominates partitioning. Pick PI to match your most-frequent join key, not the column you want to partition on. Adding PPI without an EXPLAIN-driven PI plan is a textbook way to make queries slower.

Why this still ships

I left the result honest on the dashboard. The Performance tab shows all five queries, the median timings, and the speedup column with the regressions colored in red. That's exactly what the Reliability Engineer needs to make the case for a second iteration. A perf tuning project that hides its losses is a perf tuning project that won't get a second round of investment.

What I'd do differently next time

  • Start with EXPLAIN, not intuition. Five minutes of reading plans before tuning saves an hour of regression analysis.
  • Stats first, structure second. Stats are reversible (just re-collect). PI changes mean recreating the table.
  • If you must change PI, change it for the join you actually have. Our query mix is dominated by customer_sk joins; PI on (customer_sk, sold_date_sk) with PPI on sold_date_sk is a more defensible config than what I chose.
  • Track perf history, not just current numbers. The DW_DQ.perf_log table makes "did this regress?" a one-line query.

What SDD actually bought us

The temptation with a methodology post is to over-claim. SDD didn't make the build faster than ad-hoc would have. What it did was:

  • Settled arguments before they happened. "Should anomaly rows be dropped or tagged?" — roadmap.md Phase 1 said tag. Done. No meeting.
  • Kept scope honest. mission.md said non-goals: SaaS infrastructure, reporting tool. The dashboard is a thin PHP page reading from a FastAPI helper — not a BI product. The pipeline is local + Docker — not a managed service.
  • Made personas the unit of design review. Three personas → three dashboard tabs (Reliability, Silver/Gold for the BA, Performance for whoever owns the tuning). When I added a fourth tab, I had to justify it against an existing persona.
  • Forced demoable phases. roadmap.md said "each phase is shippable and demo-able on its own." That's why Bronze got its own dashboard tab before Silver was a single SQL line. Each phase landed with a thing to show.
  • Made the negative result publishable. The Phase 3 spec required showing improvement on the dashboard. It didn't say "if there is improvement." Without that line, the regressions would have quietly disappeared. With it, they're the most useful artifact in the repo.

The four spec files added up to fewer lines than this article. They didn't decide everything — they couldn't have predicted the FastLoad-per-call session bug or the CHAR(1) padding trap. But they decided the things that matter most: what is being built, why, for whom, with what tools, in what order, and how do we know it works. With those settled, the rest is just engineering.

If you take one thing away from the SDD framing: write the four spec files before you open your IDE. Keep them under 200 lines combined. Reference them at every contentious decision. They won't write your code — but they'll tell you when you're about to write the wrong code.
